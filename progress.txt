## Issue #21 — schema-drift guard between packages/schema and ops's vendored copy
- Urgency raised because PR #40 (open, adds `brand.motion`) is the first real
  canonical schema change since the freeze; no mechanism existed to catch a
  missed ops re-sync.
- ops repo isn't accessible from here, so per the issue's own fallback plan
  I built a vendored shape-snapshot mechanism entirely within this repo:
  - `packages/schema/src/shape.ts`: normalizes a Zod schema into a
    comparable, JSON-serializable shape via `_def` introspection (field
    keys/nesting/kind — object/array/record/enum/literal/primitive),
    deliberately dropping check-level detail (regex, min/max) that
    legitimately differs from ops's hand-converted `.mjs` copy. Also
    `presetsShape()` for `presets.ts`'s plain id lists/token maps, and a
    generic `diffShapes()` deep-diff.
  - `packages/schema/ops-shape.snapshot.json`: frozen snapshot of the shape,
    representing the last known-synced `ops/lib/schema`. Generated by
    `pnpm schema:snapshot-ops` (packages/schema/scripts/refresh-ops-snapshot.ts,
    root package.json script since tsx only lives at root, matching
    new-client/eject-client's pattern).
  - `packages/schema/src/ops-drift.test.ts`: 3 vitest tests — ClientConfig
    shape matches snapshot, presets shape matches snapshot, and a self-test
    that seeds a `brand.motion`-style field-only-here-not-in-snapshot drift
    and asserts diffShapes catches it (the DoD's fixture requirement).
  - Wired into the gate for free: `ralph/gate.sh` already runs
    `turbo run test`, which now picks up the schema package's new test file
    — no ci.yml changes needed.
  - Documented in AGENTS.md § "Contracts this repo owns": what the guard
    does, how to refresh the snapshot, and the rule to sync ops first rather
    than regenerate the snapshot to silence a real drift.
- Verified end-to-end: manually deleted a field from the live
  ops-shape.snapshot.json, confirmed the test failed with a readable diff +
  re-sync hint, then restored it and confirmed green again.

Issue #79: fix(template) hero unreadable on junk-removal preset — bg-fg × text-on-primary token mispairing

What shipped (smallest complete slice, packages/template only — no packages/schema
contract change, so no ops re-sync required):
- Root cause: Hero.astro's `bg-fg text-on-primary` paired `--brand-fg` (surface)
  against `--brand-on-primary` (text) — a token designed to pair with `--brand-
  primary`, not `--brand-fg`. junk-removal resolves to #161616 bg vs #0d0d0d
  text, ~1.07:1 (WCAG needs 4.5:1). The other 3 presets only looked fine because
  their on-primary happens to be #ffffff.
- Fix: introduced `--color-on-fg` (theme.css `@theme`), mapped to a new
  `--semantic-color-content-inverse: bg` line in brand-overlay.ts's
  brandOverlayVars(). Reuses `bg` (not a new schema field) because fg/bg is
  already the page's base body-text/background pairing — pairing hero text
  with `bg` when `fg` is used as the hero surface is just that same pairing
  read in reverse, so it's contrast-correct by construction, per-preset, and
  per-override, with zero new PaletteTokens surface (avoids an ops/lib/schema
  contract-drift sync — packages/schema untouched).
- Hero.astro: `bg-fg text-on-primary` -> `bg-fg text-on-fg`; same swap on the
  heroSub paragraph, the "Call {phone}" button's border/text/hover, and the
  "Serving ..." line. Left the CTA button's `bg-primary text-on-primary`
  untouched — that pairing was already correct (issue said so explicitly).
- New packages/template/src/lib/contrast.ts: dependency-free WCAG relative-
  luminance + contrastRatio + meetsAAContrast (3- and 6-digit hex). 8 unit
  tests incl. a fixture asserting junk-removal's exact pre-fix pair
  (#161616 vs #0d0d0d) is correctly flagged red (~1.07:1) and its post-fix
  pair (#161616 vs #ffffff) passes.
- checkClientAcceptance (acceptance.ts) gained an unconditional (not gated by
  realData — this is structural, not fake-intake-data) contrast guardrail:
  resolvePalette(config) checked against WCAG AA 4.5:1 for primary/on-primary
  (`low-contrast-cta`) and fg/bg (`low-contrast-hero` — covers the hero since
  on-fg now == bg). Runs for every preset and any cssVarOverrides, so a future
  preset or override can't regress this silently. 4 new acceptance.test.ts
  cases: all 4 shipped presets pass both pairs; a bad on-primary override
  flags low-contrast-cta; a bad fg/bg override flags low-contrast-hero.
- Verified against built output, not just unit tests: rebuilt all preview
  apps, grepped dist/index.html — junk-removal's --semantic-color-content-
  inverse now resolves to #ffffff (was implicitly using on-primary #0d0d0d
  via text-on-primary); the other 3 presets resolve on-fg to their own bg
  (near-white tint, e.g. #f7f8f3) instead of a flat #ffffff — an
  imperceptible, more-correct shift, not a regression.
- Gate green: pnpm build + pnpm check + turbo test (all suites incl. the
  schema-drift guard, untouched since packages/schema wasn't touched).
